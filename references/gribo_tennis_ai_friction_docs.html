Tennis image analysis friction test

A Gribo Digital experiment for measuring how much friction appears when using AI to analyze tennis practice images: racket position, body posture, court context, stroke phase and actionable coaching notes.

Core question

Can AI turn casual tennis photos into structured, useful and repeatable technical feedback without needing a full computer-vision pipeline from day one?

Quickstart

The goal of this test is not to build a finished coaching product yet. The first goal is to identify the friction points: image quality, prompt stability, output structure, tennis-specific accuracy and how much manual review is still needed.

1

Collect a small image set

Use 8–15 tennis practice images from similar sessions: forehands, backhands, serves, ready position and court movement.

2

Run the same prompt repeatedly

Ask the model to return a structured technical analysis, then compare consistency across images and repeated runs.

3

Score friction, not perfection

Track where the workflow slows down: unclear image, vague feedback, hallucinated technique, missing tennis context or manual formatting.

What we are testing

This page documents a practical AI friction test: how far a general vision model can go when analyzing real tennis images, and what infrastructure is needed before this becomes a reliable internal tool.

Vision reasoning

Can the model identify posture, racket angle, court positioning and probable stroke phase from a single image?

Coaching usefulness

Can the model produce feedback that feels specific enough to act on during the next practice session?

Output repeatability

Does the same image produce similar observations when analyzed again, or does the diagnosis drift too much?

Workflow friction

How many manual steps are needed to upload, analyze, clean, store and compare results?

Analysis pipeline

The first version should stay lightweight. No custom model, no full pose-estimation stack, no app UI. The test should validate whether the workflow deserves deeper engineering.

friction-test.pipeline 
input: tennis_practice_image

steps:
  - normalize_image_context
  - identify_visible_body_markers
  - infer_stroke_phase
  - evaluate_racket_path
  - evaluate_footwork_and_balance
  - generate_coaching_notes
  - score_confidence_and_friction

output:
  format: structured_json + readable_summary
  review_required: true
Stage Purpose Friction risk
Image intake Store photo, angle, session date and shot type if known. Missing metadata makes comparisons harder.
Visual analysis Detect stance, racket face, body rotation and court relation. Single-frame inference can overclaim movement.
Technical feedback Translate visual observations into training suggestions. Feedback can become generic if the prompt is weak.
Review Human checks if the advice is fair, useful and safe. Manual review may become the bottleneck.

Analysis schema

To avoid vague outputs, every image should return the same fields. This makes it easier to compare sessions and eventually build a searchable project archive.

analysis-result.json 
{
  "image_id": "tennis-session-001",
  "shot_type": "forehand | backhand | serve | ready_position | unknown",
  "visible_context": {

Friction metrics

The evaluation is intentionally practical. We are not measuring model intelligence in the abstract; we are measuring whether this workflow can become useful without becoming annoying to maintain.

≤ 3 Manual steps per image before analysis starts.
80% Target rate for outputs that feel specific enough to review.
< 5m Desired time from upload to saved analysis note.
Metric Good signal Bad signal
Specificity Mentions observable details from the image. Uses generic tennis advice that could apply to anyone.
Repeatability Similar analysis across repeated runs. Different diagnosis every time.
Actionability Returns one next-practice adjustment. Overloads the user with too many corrections.
Trust Clearly marks uncertainty and image limitations. Pretends to know movement patterns from one frame.

Prompt template

The prompt should force the model to separate visible evidence from inference. This matters because tennis technique involves movement, and a single image can only show part of the sequence.

prompt.md 
You are analyzing a tennis practice image.

Return structured feedback using these sections:

1. Visible evidence only
- Describe what can be directly seen in the image.
- Do not infer full motion unless the image supports it.

2. Likely stroke phase
- Choose: preparation, contact, follow-through, recovery, unknown.
- Explain confidence level.

3. Technical observations
- Stance
- Balance
- Racket position
- Body rotation
- Footwork

4. Coaching notes
- One thing to keep.
- One thing to adjust.
- One simple drill for the next session.

5. Uncertainty
- List what cannot be determined from this image.
